## Text (first comment from you, posted seconds after the submission)

```
Hi HN — I'm the author.

I built vibe-hardening after watching repo after repo generated by v0,
Lovable, Bolt, Cursor, Claude Code, Replit Agent and Windsurf ship with
the same classes of bug: OpenAI keys pushed to GitHub, Supabase
service_role JWTs embedded in React client components, Next.js API
routes with zero auth checks, NEXT_PUBLIC_*SECRET env vars leaked into
browser bundles, and npm dependencies with known CVEs the author never
looked at.

Existing tools didn't fit:
- semgrep is powerful but beginners need a ruleset
- gitleaks only does secrets
- snyk is enterprise and heavy

So I scoped a CLI specifically for vibe-coded apps:

  npx vibe-hardening scan

30+ rules across secrets, injection, auth, CORS/SSRF, Supabase RLS,
dependency CVEs (via OSV.dev), and LLM-hallucinated packages. It
fingerprints which AI tool generated your code (v0 / Lovable / Bolt /
Cursor / Claude Code / Replit Agent / Windsurf / Devin) and weights
rules accordingly. Returns a 0-100 score with an SVG badge you can put
in your README.

A few design choices I want HN to push back on:

1. The 'use client' + service_role composite rule catches real
   Supabase leaks but relies on the two strings co-existing in the
   same file. Open to smarter pass.

2. Missing-auth detection is AST-based (ts-morph) and understands
   the Vercel Cron `Bearer ${process.env.CRON_SECRET}` pattern — but
   only after one painful false positive on my own project.

3. Inline suppression is `// vibe-hardening-disable-next-line
   vh-rule-id` — ESLint style. Let me know if this is wrong.

Happy to answer anything. Full source MIT at
github.com/vibe-hardening/cli — repo opens with five-language READMEs
if that matters.

What rules would you want added?
```

## First-hour reply templates

**"Why not just use semgrep?"**
```
Fair point — semgrep is more flexible. The bet here is that a vibe
coder running 'npx vibe-hardening scan' in a fresh v0 project gets
a signal in 5 seconds without writing a config. semgrep needs
ruleset knowledge; this picks one.

For teams with security engineers, semgrep is still the right
answer. This is for the ship-on-weekend crowd.
```

**"Your rules look like ESLint rules"**
```
They are, conceptually — secret patterns + AST checks + OSV/CVE
query. The added layer is AI-platform fingerprint: v0 omits auth
in a specific way, Lovable puts service_role in client components
in a specific way. Rules weight by the detected platform.
```

**"What about false positives?"**
```
Big concern. First real dogfood on my own repo went from 36
findings to 1 actually-exploitable HIGH after I added test/
markdown/ide-internal skip logic and the Vercel Cron bearer-token
allowlist. There's an inline disable directive for the rest.

If anyone runs it on something noisy I want to know.
```

**"Isn't this just a marketing stunt?"**
```
Maybe. It caught a real missing-auth bug on my lineage-auction
side project that I'd shipped to production. If a marketing stunt
finds actual bugs, I'll take it.
```

**"What's the business model?"**
```
CLI stays MIT forever. A Pro SaaS is planned for team dashboards,
historical diff, and Slack alerts — the things a scanner in a
terminal can't do. But I'm not taking money until the OSS tool is
actually good.
```
