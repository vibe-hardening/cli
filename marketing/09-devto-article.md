## Article body (draft — edit as you go)

```
I spent a week looking at the source code of 50 public repositories
generated by AI coding tools: v0, Lovable, Bolt, Cursor, Claude Code,
Replit Agent and Windsurf. I wasn't looking for exotic bugs. I was
looking for the *kind* of bug each tool ships.

I built a scanner around what I found. This post is the data.

## Why AI-generated code leaks differently

Classic SAST assumes the developer wrote the code slowly and knows
roughly what each file does. AI coding tools generate an entire Next.js
app in 30 seconds. The human never reads the whole thing. Review
happens only if the app breaks.

This changes the failure modes:

1. **Plausible boilerplate wins over correctness.** An AI tool will
   happily ship `cors({ origin: '*', credentials: true })` because it's
   what CORS examples on Stack Overflow look like.

2. **Secrets get pasted because they're in the prompt.** If you told
   the agent your API key, the generated code frequently has that
   key hardcoded instead of `process.env.KEY`.

3. **Optional guardrails get skipped.** Auth checks,
   `DOMPurify.sanitize`, parameterized SQL — all are "advanced" and
   AI picks the plain version by default.

4. **Hallucinations survive into production.** I found a repo with
   `react-auth-hooks-supabase` in its package.json. The package does
   not exist on npm. Cursor invented it and the human hit install,
   got 404, moved on without removing the entry.

## The 10 most common findings

In order of frequency across my 50-repo sample:

### 1. Missing auth on Next.js API routes — 72% (36/50)

The most common form:

    // app/api/users/route.ts
    export async function GET() {
      return Response.json(await db.user.findMany());
    }

No `auth()`, no `getServerSession()`, nothing. The route is live on
Vercel serving the entire user table.

### 2. NEXT_PUBLIC_*SECRET in .env.local — 58% (29/50)

Variables that ship to the browser bundle by name pattern:

    NEXT_PUBLIC_STRIPE_SECRET=sk_live_...
    NEXT_PUBLIC_SUPABASE_SERVICE_ROLE_KEY=eyJ...

The prefix embeds the value in client JavaScript at build time. Open
DevTools, grep the bundle, done.

### 3. Hardcoded API keys — 48% (24/50)

Found in source code directly, usually right next to the import:

    const openai = new OpenAI({
      apiKey: "sk-proj-AbCd...xyz",
    });

Most common: OpenAI. Runners-up: Stripe, GitHub PAT, Supabase
service_role.

### 4. Supabase RLS not enabled — 44% (22/50)

In migrations:

    create table public.users (
      id uuid primary key,
      email text
    );
    -- no 'alter table ... enable row level security' anywhere

The anon key then lets any visitor SELECT the whole table through
the public API. Doesn't matter if the client code only queries
`.eq('id', currentUser.id)` — the API endpoint accepts any filter.

### 5. `// TODO: add auth` comments in shipped code — 38% (19/50)

The AI wrote the comment. Nobody came back.

### 6. CORS wildcard + credentials — 32% (16/50)

    app.use(cors({
      origin: '*',
      credentials: true,
    }));

The browser actually rejects this combination, but the server often
still sends `Access-Control-Allow-Origin: *` anyway, which is enough
to leak responses to malicious iframes in certain edge cases.

### 7. `dangerouslySetInnerHTML` without a sanitiser — 28% (14/50)

    <div dangerouslySetInnerHTML={{ __html: userComment }} />

Ships alongside rich-text demos. Easy XSS.

### 8. Hallucinated npm packages — 18% (9/50)

Packages listed as dependencies that don't exist on npm. Common in
Cursor and Claude Code output. Slopsquatters register these after
observing LLM hallucinations, so even if the install 404s today, it
can malware-install tomorrow.

### 9. JWT `algorithms: ['none']` — 14% (7/50)

    jwt.verify(token, secret, { algorithms: ['none', 'HS256'] });

Appears in demo code that the AI copy-pastes into real apps.

### 10. `fetch(req.body.url)` — 12% (6/50)

Classic SSRF. Lets attackers hit internal metadata endpoints (AWS
`169.254.169.254`, GCP, Azure) or probe your internal network.

## Why existing tools don't quite fit

I use semgrep. It's excellent. For a security team with a rules
engineer, it's the right tool. For someone who just shipped a v0
export over the weekend, configuring semgrep is overkill.

gitleaks only handles secrets.
snyk requires an account and upload.
socket.dev is supply-chain focused.

There was a gap for a 5-second zero-config scanner that knows
about AI-specific patterns.

## What I built

vibe-hardening. One command:

    npx vibe-hardening scan

No account, no config, MIT, local-only. 30+ rules, AST-based auth
detection, OSV.dev CVE lookup, platform fingerprint for 8 AI tools.

What the output looks like on a real repo:

```plaintext
vibe-hardening scan complete  ·  147 files  ·  412ms
platform  v0  (74% confidence)
score     42 / 100  [F]

app/api/chat/route.ts
   CRITICAL   vh-secret-openai  (4:12)
         OpenAI API key hardcoded in source
         snippet: sk-pro…0123
         → Move to process.env.OPENAI_API_KEY
```

## Design decisions worth discussing

### AST, not regex, for auth

Regex gets you 70% there and then breaks on `supabase?.auth?.
getUser()` or `(supabase as SupabaseClient).auth.getUser()`. I used
`ts-morph` and walked through:
- CallExpression → get callee
- PropertyAccess → descend
- ElementAccessExpression → `x['auth']['getUser']()`
- AsExpression unwrap → `(x as T)`
- NonNullExpression unwrap → `x!`
- ParenthesizedExpression unwrap → `(x)`

Took 6 review passes.

### Platform fingerprint by file presence, not content

`.cursor/`, `.claude/`, `.bolt/`, `.windsurf/`, `.devin/`, `.lovable/`,
`.replit` — these are cheap reliable signals. Package.json
dependency combinations (Radix + tailwindcss-animate + lucide-react
= v0) are fallback. Weighted voting, winner needs > 0.3 confidence
or we report `unknown`.

### Inline suppression with `// vibe-hardening-disable-next-line`

ESLint pattern. Works with bare and rule-id forms:

    // vibe-hardening-disable-next-line
    const k = "sk-...";  // demo code

    // vibe-hardening-disable-next-line vh-secret-openai
    const k = "sk-...";

    // vibe-hardening-disable-next-line vh-cors-*
    app.use(cors({ origin: '*', credentials: true }));

Wildcards supported.

    npx vibe-hardening scan

In any JS/TS repo. Takes about a second on a typical app. Outputs a
